## Section 1: Panel Layout (Item 1)

### Problem

Preview and lineage are both views inside a single `dbtCoreToolsPanel` container. They should each have their own container so they appear as separate tabs in the Panel area.

### Design

Split `viewsContainers.panel` in `package.json` from one container into two:

- `dbtLineagePanel` — contains `dbtCoreTools.lineageView`
- `dbtPreviewPanel` — contains `dbtCoreTools.previewView`

Each gets its own title and icon. No code changes — the `WebviewViewProvider` registrations in `extension.ts` register by view ID, not container ID.

**package.json change:**

```json
"viewsContainers": {
  "panel": [
    {
      "id": "dbtLineagePanel",
      "title": "dbt Lineage",
      "icon": "$(type-hierarchy)"
    },
    {
      "id": "dbtPreviewPanel",
      "title": "dbt Preview",
      "icon": "$(table)"
    }
  ]
},
"views": {
  "dbtLineagePanel": [
    {
      "type": "webview",
      "id": "dbtCoreTools.lineageView",
      "name": "dbt Lineage"
    }
  ],
  "dbtPreviewPanel": [
    {
      "type": "webview",
      "id": "dbtCoreTools.previewView",
      "name": "dbt Preview"
    }
  ]
}
```

## Section 2: Lineage Rewrite with React Flow (Items 2-8)

### Problem

The current D3 + ELK lineage webview lacks drag, expand/collapse toggle, zoom stability on click, and visual polish (node contrast, selected node highlight, boundary-aware expand buttons). Seven feedback items all touch this surface.

### Design

Replace the D3 + ELK rendering layer with React Flow + dagre. The extension-side data flow (`buildGraphData()` → `postMessage` → webview renders) stays the same; only the webview rendering changes.

#### Build Setup

- Add `react`, `react-dom`, `@xyflow/react`, and `dagre` as devDependencies
- Add a second esbuild entry point: `src/features/lineage/webview/App.tsx` → `dist/lineage.js` (JSX, browser platform)
- Lineage `index.html` loads `dist/lineage.js` instead of vendored D3/ELK
- Remove vendored `d3.min.js` and `elk.bundled.js` from the lineage webview
- Remove `d3` and `elkjs` from devDependencies

#### React Flow Graph Component

- Dagre layout engine, left-to-right direction (same flow as current)
- Custom node component with: name, materialization label, test count badge, contract badge
- Node colors: same resource-type palette but tuned for better dark-background contrast (brighter/more saturated)
- Selected/current node: distinct highlight (brighter border or glow)

#### Interactions (Built-in with React Flow)

- **Drag:** nodes draggable by default
- **Zoom:** scroll wheel only, no zoom on click (React Flow default)
- **Pan:** click-drag on background

#### Expand/Collapse Toggle (Item 4)

- Each node has directional handle buttons (upstream on left edge, downstream on right edge)
- First click expands (+): adds 1-depth neighbors via existing `mergeGraph` flow, button changes to `−`
- Second click collapses (−): removes the expanded nodes, button reverts to `+`
- Track expanded state per node+direction in the webview

#### Boundary-Aware Handles (Item 5)

- Add `hasUpstream` and `hasDownstream` booleans to `GraphNode` data sent from the extension
- Check `parentMap` and `childMap` in `buildGraphData()`: if a node has no non-test parents, `hasUpstream = false`; same for children/downstream
- Webview hides the corresponding expand button when false

#### Lock + Reset (Item 8)

- Lock toggle defaults to **checked** (locked)
- Add a "Reset" button next to the lock toggle
- Reset re-centers the graph on the current editor's model while keeping the lock on
- Implementation: "Reset" sends a message to the extension requesting current editor's node; extension responds with an `updateCenter` that bypasses the lock check (new message type `resetCenter` or a flag on `updateCenter`)

#### Source Expansion Bug (Item 9a)

- Investigate whether source `unique_id`s appear in `parentMap`/`childMap` from the manifest
- If not, build expansion for sources from the `depends_on` fields on child nodes (reverse lookup)
- Ensure `buildGraphData()` can expand both upstream and downstream from source nodes

#### Zoom Stability (Item 7)

- React Flow handles this natively — zoom only responds to scroll wheel, not click events
- No custom work needed

## Section 3: Command Execution (Items 10-13)

### Problem

Four issues: the two-step options picker is redundant, `uv run dbt` fails because the shell can't find `uv`, `dbt show` fails with a relative `dbtCommand` path, and the default show limit is too low.

### Design

#### Options Picker Simplification (Item 10)

Remove the two-step flow in `showOptionsPicker()`. Go straight to the multi-select quick pick:

- Show the multi-select with scope options (upstream, downstream, full-refresh)
- **Enter with nothing selected** → run vanilla command (return `{}`)
- **Enter with selections** → run with those options
- **Escape** → cancel entirely (return `undefined`)

This already matches `showQuickPick` with `canPickMany: true` behavior: Escape returns `undefined`, Enter with empty selection returns `[]`.

#### dbtCommand PATH / venv Resolution (Item 11)

Instead of fixing PATH propagation for `uv run dbt`, detect and use the project's venv directly:

1. When resolving `dbtCommand`, check for `.venv/bin/dbt` (or `.venv/Scripts/dbt.exe` on Windows) relative to the project root
2. If found and `dbtCommand` is the default `"dbt"`, use the absolute venv path automatically
3. Also check the `VIRTUAL_ENV` environment variable as a fallback
4. If `dbtCommand` is explicitly set to something non-default, respect the user's override

This sidesteps the PATH problem entirely — calling an absolute path to a real executable needs no shell resolution. Zero configuration for the common case (uv-managed dbt project with a `.venv`). This also fixes item 12 (`dbt show` failing with relative paths) since the resolved path is always absolute.

Additionally, pass `process.env` explicitly to both execution paths as a safety net:
- `executeAndCapture`: add `{ shell: true, env: process.env }` to the `execFile` call
- `executeInTerminal`: pass `{ env: process.env }` in `ShellExecution` options

#### Show Limit Default (Item 13)

Change `dbtCoreTools.showLimit` default from `5` to `100` in `package.json`.

## Section 4: Compiled SQL Loading Indicator (Item 14)

### Problem

The existing `withProgress` notification toast and inline `-- Compiling...` placeholder text are not noticeable enough.

### Design

Show a spinner in the manifest status bar item while compiling, consistent with the existing `$(sync~spin)` pattern used during `dbt parse`.

When `dbt compile` starts:
- Manifest status bar shows `$(sync~spin) compiling {modelName}...`
- The existing inline placeholder text (`-- Compiling...`) remains in the virtual document

When complete:
- Status bar reverts to normal `parsed: {relativeTime}` display
- Virtual document updates with compiled SQL

Keep the `withProgress` notification as well — having both the status bar spinner and the notification gives multiple visual cues.

## Section 5: Icon and Button Cleanup (Items 15, 16)

### Problem

The extension needs the official dbt logo, and the lineage button in the editor title bar is redundant now that lineage has its own panel.

### Design

#### Better Icon (Item 15)

Replace `media/dbt-icon.svg` with the official dbt logo (orange hexagon "d" mark). Used for the `jinja-sql` language icon in the file explorer.

#### Remove Lineage Button (Item 16)

Remove the `dbtCoreTools.showLineage` entry from `menus.editor/title` in `package.json`. Keep the command registered so it remains accessible via Command Palette.

## Section 6: Manifest Error Handling (Item 9b — error spam)

### Problem

Repeated `[error] Failed to load manifest for kipptaf: SyntaxError: ...` messages when `dbt parse` or `dbt compile` writes a partial `manifest.json` and the file watcher triggers `reloadManifest()` mid-write.

### Design

1. **Debounce the file watcher:** When the manifest file watcher fires, wait 500ms before calling `reloadManifest()`. If another change event arrives during the window, reset the timer. This lets dbt finish writing before we attempt to parse.

2. **Downgrade transient errors:** Change the log level for JSON parse failures in `reloadManifest()` from `[error]` to `[warn]`, since they are transient and self-resolving once dbt finishes writing.
