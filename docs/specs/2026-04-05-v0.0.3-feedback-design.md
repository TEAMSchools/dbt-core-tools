## Group 1: Executor & Command Runner

### Bug #10 ŌĆö Defer flags sent when defer is off

**Root cause:** `getCommandOptions()` in `src/commands/modelCommands.ts` reads `deferManifestPath` directly from settings and always returns it as `deferState`. It never checks `DeferToggle.isDeferred()`.

**Fix:** Import `getDeferToggle()` from `src/extension.ts`. Only return `deferState` when the toggle reports `isDeferred(projectName) === true`. When defer is off, return `deferState: undefined`.

**Files:** `src/commands/modelCommands.ts`

### #11 ŌĆö Always show options picker

**Current behavior:** Each command (run, build, test) has two variants: plain and with-options. The editor title bar uses submenus with both variants.

**New behavior:** Each command always shows a two-step quick pick:

1. **First pick** ŌĆö "Run" (plain, no options) vs "Run with options..."
2. **Second pick** (only if "with options" chosen) ŌĆö multi-select: Full Refresh, Upstream, Downstream. For `test`, only show Upstream and Downstream (Full Refresh is not valid for test).

Pressing Escape at either step cancels.

**Changes:**
- Remove `runModel`, `buildModel`, `testModel` exports from `modelCommands.ts`. Remove `runModelOptions`, `buildModelOptions`, `testModelOptions` ŌĆö replace all six with three: `runModel`, `buildModel`, `testModel`, each always showing the two-step picker.
- `showOptionsPicker(subcommand)` takes the subcommand so it can filter options (no Full Refresh for test).
- Remove submenus from `package.json` ŌĆö run/build/test become single buttons in editor title bar.
- Remove the `*Options` command registrations from `package.json` and `extension.ts`.
- Context menu items also go through the picker.

**Files:** `src/commands/modelCommands.ts`, `src/commands/optionsPicker.ts`, `package.json`, `src/extension.ts`

### #4 ŌĆö Command queuing

**Current behavior:** `executeInTerminal()` shows a warning if a terminal for the project is already open.

**New behavior:** Queue commands per project. Reuse the same terminal.

**Implementation:**
- Add a per-project command queue: `Map<string, string[]>`.
- When `executeInTerminal` is called and a terminal is already running, push the command onto the queue instead of showing a warning.
- Use VS Code's `Terminal.shellIntegration` API (`onDidEndTerminalShellExecution`) to detect when a command finishes. On completion, check the queue and `sendText` the next command.
- Fallback for terminals without shell integration: use `onDidCloseTerminal` as today (queue drains on terminal close).

**Files:** `src/core/executor.ts`

## Group 2: Lineage

### #1 ŌĆö Persistent bottom panel

**Current behavior:** Lineage opens as a `WebviewPanel` in the editor area (ViewColumn.Two). Must be explicitly opened. Closes when tab is closed.

**New behavior:** Lineage is a `WebviewViewProvider` registered to the VS Code Panel area (alongside Terminal, Problems). Persistent ŌĆö always available in the panel. Updates automatically when the active editor changes.

**Implementation:**
- Register a `views` contribution in `package.json` under a new view container in the `panel` location:
  ```json
  "viewsContainers": {
    "panel": [{
      "id": "dbtCoreToolsPanel",
      "title": "dbt Core Tools",
      "icon": "$(type-hierarchy)"
    }]
  },
  "views": {
    "dbtCoreToolsPanel": [{
      "type": "webview",
      "id": "dbtCoreTools.lineageView",
      "name": "Lineage"
    }]
  }
  ```
- New class `LineageViewProvider implements WebviewViewProvider` in `src/features/lineage/lineagePanel.ts`, replacing the current `_panel` approach.
- `resolveWebviewView()` sets up webview HTML, message handling, and stores the view reference.
- `showLineage` command becomes `dbtCoreTools.lineageView.focus` (built-in VS Code command to reveal the panel).
- `onDidChangeActiveTextEditor` calls the provider's update method.
- Graph building logic (`buildGraphData`), webview message handling, and webview assets (HTML/JS/CSS) remain the same.
- Wire manifest change events to refresh lineage view (same as compiled SQL wiring).

**Files:** `src/features/lineage/lineagePanel.ts`, `package.json`, `src/extension.ts`

### Bug #6 ŌĆö Lineage is blank

**Root cause 1 ŌĆö Race condition:** The panel is created, HTML is set, and `postMessage` is called immediately. The webview loads D3 and dagre from CDN before registering its message listener. The `postMessage` fires before the webview is ready ŌĆö graph data is lost.

**Root cause 2 ŌĆö CDN dependency:** D3 and dagre are loaded from `cdn.jsdelivr.net`. In environments without external network access (e.g., Codespaces behind a firewall), the scripts never load and nothing renders.

**Fix 1 ŌĆö Ready handshake:** Webview posts `{ type: "ready" }` after scripts load and message listener is registered. Extension buffers graph data until "ready" is received. On "ready", sends the buffered data.

**Fix 2 ŌĆö Bundle D3 + dagre locally:** Install `d3` and `@dagrejs/dagre` as devDependencies. Copy their minified builds to `src/features/lineage/webview/vendor/`. Reference them via local `<script>` tags instead of CDN URLs. Remove CDN from CSP policy.

**Fix 3 ŌĆö Empty states:** When manifest is null, send a message to the webview to show "No manifest found ŌĆö run dbt parse". When no model is found for the active file, show "No dbt model found for this file".

**Files:** `src/features/lineage/lineagePanel.ts`, `src/features/lineage/webview/main.js`, `src/features/lineage/webview/index.html`, `package.json`

## Group 3: Compiled SQL, Show, & Defer Path

### Bug #2 ŌĆö Defer state path broken

**Root cause:** `deferManifestPath` stores a relative path. `buildDbtCommand` passes it as `--state=<value>` directly. dbt resolves `--state` relative to the current working directory, but the terminal's cwd may not match the workspace root.

**Fix:** In `getCommandOptions()`, resolve the defer manifest path to an absolute path by joining it against the workspace folder root (`vscode.workspace.workspaceFolders[0].uri.fsPath`) before returning it as `deferState`.

**Files:** `src/commands/modelCommands.ts`

### Bug #7 ŌĆö Compiled SQL not working

**Root cause:** `dbt parse` does not populate `compiled_code` in the manifest. Only `dbt compile` or `dbt run` does. The node exists but `compiled_code` is null.

**Fix ŌĆö Auto-compile on demand:** When `showCompiledSql` finds `compiled_code` is null, automatically run `dbt compile -s <model>` via `executeAndCapture()`, wait for it to finish, reload the manifest, then read and display the compiled SQL.

**Fix ŌĆö Compile on save:** When a `.sql` file is saved and a compiled SQL virtual document is open for that model, run `dbt compile -s <model>` in addition to the regular `dbt parse`. The manifest watcher then triggers the virtual document refresh, keeping the compiled SQL panel in sync. This only fires when a compiled SQL document is open ŌĆö otherwise `parseOnSave` runs `dbt parse` as before.

**Fix ŌĆö Error message:** Change the error text from "Run dbt parse to compile the project" to "Compiling..." (since auto-compile will be in progress).

**Files:** `src/features/compiledSql.ts`, `src/features/parseOnSave.ts`

### Bug #13 ŌĆö Show does nothing

**Root cause:** `executeAndCapture` runs `dbt show` and captures stdout. If the command succeeds but stdout contains no pipe-delimited lines (e.g., only log messages), `parseDbtShowOutput` returns `{ columns: [], rows: [] }`. The webview receives a `results` message with empty data and renders nothing.

**Fix:** After parsing, if `columns.length === 0`, treat it as an error. Display the full stdout + stderr in the error panel so the user can see what happened. Also check if the result is related to the defer path issue (#2) and surface that clearly.

**Files:** `src/features/preview/previewPanel.ts`

## Group 4: Navigation & Profiles

### Bug #5 ŌĆö Macro go-to-definition resolves wrong file

**Root cause:** The definition provider matches `macro.name` against the word at cursor and returns the first match. For package macros, `original_file_path` is relative to the package install directory (e.g., `macros/sql/union.sql` from `dbt_utils`), and `safeJoinPath(project.rootPath, macro.original_file_path)` may construct a path that doesn't exist (the actual location is under `dbt_packages/dbt_utils/macros/sql/union.sql`). The file opens but doesn't exist.

**Fix:**
1. For qualified references like `dbt_utils.union_relations`, parse the full `package.macro_name` pattern and match against both `macro.package_name` and `macro.name`.
2. For unqualified references, prefer the current project's macros over package macros.
3. Use the macro's `unique_id` to derive the correct path. The `unique_id` format is `macro.<package>.<name>`. For non-project macros, resolve via `dbt_packages/<package>/<original_file_path>`.
4. Add `fs.existsSync` check before navigating. Show a warning if the file doesn't exist: "Macro file not found. Run dbt deps to install packages."

**Files:** `src/features/definition.ts`, `src/utils/patterns.ts` (add qualified macro pattern)

### Bug #9 ŌĆö No targets found for profile

**Root cause:** `selectTarget` in `src/statusbar/targetSelector.ts` passes `project.name` as the profile name to `parseProfileTargets`. But the profile name in `profiles.yml` is set via the `profile:` key in `dbt_project.yml`, which doesn't have to match the project name.

**Fix:** Read the `profile:` key from the project's `dbt_project.yml` and use that as the lookup key. Add a `profileName` property to `DbtProject` that's extracted from `dbt_project.yml` during project construction (alongside the existing `name` extraction). Use `project.profileName` in `selectTarget`.

**Files:** `src/core/project.ts`, `src/statusbar/targetSelector.ts`

### #8 ŌĆö Properties YAML location

**Current behavior:** Properties `.yml` is scaffolded in the same directory as the `.sql` file.

**New behavior:** Controlled by a new setting `dbtCoreTools.propertiesLocation`:
- `"folder"` (default) ŌĆö creates in a sibling `properties/` subdirectory (e.g., `models/staging/properties/stg_model.yml`). Creates the `properties/` directory if it doesn't exist.
- `"inline"` ŌĆö creates in the same directory as the `.sql` file (current behavior).

**Files:** `src/features/properties.ts`, `package.json` (new setting)

## Group 5: Polish

### #3 ŌĆö Jinja syntax highlighting

- Add `samuelcolvin.jinjahtml` to `extensionDependencies` in `package.json`.
- Add `configurationDefaults` to `package.json`:
  ```json
  "configurationDefaults": {
    "files.associations": { "*.sql": "jinja-sql" }
  }
  ```
- `updateContextKeys` in `src/extension.ts`: also treat `jinja-sql` language ID as a SQL file when setting `isDbtSqlFile` context key.

Document selectors for definition/hover/completion providers already include both `sql` and `jinja-sql`.

**Files:** `package.json`, `src/extension.ts`

### #12 ŌĆö Button tooltips

Add extension name prefix to submenu labels and button tooltips in `package.json`. Since submenus are being removed (#11), this applies to the remaining single-button commands:
- Run: "dbt Core Tools: Run Model"
- Build: "dbt Core Tools: Build Model"
- Test: "dbt Core Tools: Test Model"
- Show: "dbt Core Tools: Show Model"
- Stage External: "dbt Core Tools: Stage External Sources"
- Lineage: "dbt Core Tools: Show Lineage"
- Compiled SQL: "dbt Core Tools: Show Compiled SQL"
- Toggle Properties: "dbt Core Tools: Toggle SQL/Properties"

These are already the `title` values on the commands, which VS Code uses as tooltips in the editor title bar.

**Files:** `package.json` (verify titles are correct)

### #14 ŌĆö Error logging

Create an `OutputChannel` named `"dbt Core Tools"` during activation. Export a `getOutputChannel()` getter from `src/extension.ts`. Log errors in:
- Command execution failures (`executor.ts`)
- Manifest load/parse errors (`project.ts`)
- File-not-found in definition provider (`definition.ts`)
- Compile failures (`compiledSql.ts`, `parseOnSave.ts`)
- Show command failures (`previewPanel.ts`)

No verbose/debug logging. Errors only.

**Files:** `src/extension.ts`, `src/core/executor.ts`, `src/core/project.ts`, `src/features/definition.ts`, `src/features/compiledSql.ts`, `src/features/parseOnSave.ts`, `src/features/preview/previewPanel.ts`

### #15 ŌĆö Compiled SQL icon

Change `showCompiledSql` icon from `$(split-horizontal)` to `$(file-code)`.

**Files:** `package.json`

### #16 ŌĆö Stage external icon

Change `stageExternalSources` icon from `$(database)` to `$(open-in-product)`.

**Files:** `package.json`
