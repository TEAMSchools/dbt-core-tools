# v0.0.4 Feedback Design Spec

**Issue:** [TEAMSchools/dbt-core-tools#5](https://github.com/TEAMSchools/dbt-core-tools/issues/5)
**Date:** 2026-04-05

## Overview

Fifteen feedback items from v0.0.4 testing, grouped into six sections: path resolution, autocomplete, file icons, lineage overhaul, manifest status, and preview panel.

---

## Section 1: Path Resolution (Items 1-5, 14)

### Problem

`profilesDir` is passed as-is to dbt commands. When set to a relative path like `.dbt`, it fails in:

- Background `spawn`/`execFile` calls (parseOnSave, compiledSql) where cwd may not be workspace root
- `dbt show` via `executeAndCapture`
- Target selector's `parseProfileTargets` lookup

All five error messages trace to the same root cause: `--profiles-dir=.dbt` not resolving to an absolute path.

### Design

Resolve `profilesDir` to absolute at runtime, the same way `deferManifestPath` is already resolved in `modelCommands.ts:getCommandOptions()`. If the value is relative, resolve it against the workspace root folder.

**Call sites to update:**

- `src/commands/modelCommands.ts` — `getCommandOptions()` (already resolves `deferManifestPath`, add same for `profilesDir`)
- `src/features/parseOnSave.ts` — `handleSave()` reads `profilesDir` from config
- `src/features/compiledSql.ts` — `showCompiledSql()` reads `profilesDir` from config
- `src/statusbar/targetSelector.ts` — `selectTarget()` reads `profilesDir` from config

**No settings changes.** The user-facing setting value stays as-is (e.g. `".dbt"`). Resolution is purely at runtime before passing to dbt.

**Pattern:**

```typescript
const wsRoot = vscode.workspace.workspaceFolders?.[0]?.uri.fsPath ?? "";
const resolved =
  profilesDir && !path.isAbsolute(profilesDir)
    ? path.resolve(wsRoot, profilesDir)
    : profilesDir;
```

Since this pattern is used in 4+ places, extract a shared helper `resolveWorkspacePath(relativePath: string): string` in `src/commands/modelCommands.ts` (co-located with `getCommandOptions()` which already does this for `deferManifestPath`). All call sites import and use this helper.

## Section 2: Autocomplete Fixes (Item 7)

### Problem

Three issues:

1. `ref` and `source` don't appear as completions after `{{ `
2. Completions break after typing a space (`{{ ` stops triggering)
3. Installed package macros don't appear (likely same root cause as #2)

### Design

**Trigger characters:** Add `"{"` to the trigger character list in the `registerCompletionItemProvider` call in `extension.ts`. Currently only `"("` is registered, so VS Code never re-invokes the provider when typing `{{`.

```typescript
// Before:
vscode.languages.registerCompletionItemProvider(
  dbtDocumentSelector,
  new DbtCompletionProvider(),
  "(",
);
// After:
vscode.languages.registerCompletionItemProvider(
  dbtDocumentSelector,
  new DbtCompletionProvider(),
  "(",
  "{",
);
```

**Built-in functions:** When the `{{ ` pattern matches, return dbt built-in functions alongside manifest macros:

- `ref`, `source`, `config`, `var`, `env_var`, `log`, `return`, `is_incremental`, `this`, `adapter.dispatch`, `adapter.quote`, `set`, `do`, `if`, `for`, `macro`, `block`

These are returned as `CompletionItem` entries with `CompletionItemKind.Function` and a detail label like `"dbt built-in"` to distinguish them from package macros.

**Files changed:** `src/features/completion.ts`, `src/extension.ts` (one line).

## Section 3: File Icon (Item 8)

### Problem

SQL files in the explorer use a generic icon. User wants the dbt logo.

### Design

Use the VS Code `languages` contribution point to declare a default icon for the `jinja-sql` language ID. This is the same mechanism dbt Power User uses.

**package.json addition:**

```json
"contributes": {
  "languages": [
    {
      "id": "jinja-sql",
      "icon": {
        "light": "./media/dbt-icon.svg",
        "dark": "./media/dbt-icon.svg"
      }
    }
  ]
}
```

**New file:** `media/dbt-icon.svg` — dbt logo SVG.

This icon acts as a fallback. It only appears when the user's active icon theme doesn't already define a specific icon for `jinja-sql` files. It does not replace the user's icon theme.

## Section 4: Lineage Overhaul (Items 9, 10, 11, 12, 13)

### Item 9: Switch from dagre to ELK

**Problem:** Dagre's Sugiyama layout spreads nodes too far apart, especially with fan-in/fan-out patterns common in dbt lineage.

**Design:** Replace dagre with [elkjs](https://github.com/kieler/elkjs) in the webview. Use ELK's `layered` algorithm with compaction enabled for significantly denser layout.

Changes:

- Vendor `elk.bundled.js` into `src/features/lineage/webview/vendor/` (replaces `dagre.min.js`)
- Update `main.js` to use ELK's async API (`elk.layout()` returns a Promise)
- Update `index.html` script tags and CSP
- Remove `dagre.min.js` from vendor
- Keep `d3.min.js` for SVG rendering (ELK only does layout, not rendering)
- ELK layout options: `algorithm: "layered"`, `direction: "RIGHT"`, with compact node/edge spacing

### Item 10: Tests as badges

**Problem:** Test nodes appear as separate downstream nodes, cluttering the graph.

**Design:** Filter test resource types out of the graph entirely. Instead, count tests per model by scanning `child_map` for entries starting with `test.`, and include a `testCount` number on `GraphNode`.

Changes:

- `lineagePanel.ts:buildGraphData()` — skip nodes where `resourceType === "test"` during BFS traversal; after building the graph, count test children for each node and attach as `testCount`
- `GraphNode` interface — add `testCount: number`
- `main.js` — render a small badge (beaker codicon + count) on nodes where `testCount > 0`

### Item 11a: Expand/collapse buttons

**Problem:** The expand/collapse triangle handles at node edges are small and easy to miss.

**Design:** Replace the triangle `polygon` elements with circular `+`/`-` buttons:

- Small circle (r=10) with a `+` or `-` glyph centered inside
- Positioned at the right edge (downstream) and left edge (upstream) of each node
- Show `+` when there are unexpanded neighbors in that direction, `-` when already expanded
- Use the node's border color for the circle stroke, theme foreground for the glyph
- Slightly larger click target than the current triangles

### Item 11b: Expand preserves selected node

**Problem:** Clicking an expand handle on a neighbor node changes `_currentNodeId` to that neighbor, shifting the "selected" highlight.

**Design:** Change the expand flow:

- Extension side: when handling the `expand` message, rebuild graph data centered on the expanded node but send the result as a `mergeGraph` message type (new) with the original `_currentNodeId` preserved
- Webview side: on `mergeGraph`, add new nodes/edges to the existing `_graphData` (deduplicating by node ID and edge key), keep `_currentNodeId` unchanged, then re-render

This way the graph grows outward from the expanded node while the visual selection stays on the original model.

### Item 12: Source files show lineage

**Problem:** Opening a source's YAML file clears the lineage panel because `_getActiveNodeId()` only searches `getNodes()`, not `getSources()`.

**Design:** Add a `findSourceByFilePath()` method to `DbtProject` (mirrors `findNodeByFilePath()` but searches `getSources()`). In `_getActiveNodeId()`, fall back to source lookup when node lookup returns null.

### Item 13: Rename panel

**Problem:** Panel is labeled "Lineage"; should be "dbt Lineage".

**Design:** Change `"name": "Lineage"` to `"name": "dbt Lineage"` in `package.json` views contribution.

## Section 5: Manifest Relative Time (Item 6)

### Problem

Status bar shows absolute time like `parsed: 14:32`. User wants relative time.

### Design

Replace the `HH:MM` format with relative strings: "just now" (< 1m), "1m ago", "5m ago", "2h ago", "1d ago", etc.

Since the mtime is a snapshot, the display goes stale. Add a `setInterval` (every 30 seconds) that re-calls `update()` to keep the relative time fresh. Dispose the interval in `dispose()`.

**Helper function:** `formatRelativeTime(date: Date): string` — pure function, easy to unit test.

**Files changed:** `src/statusbar/manifestStatus.ts`.

## Section 6: Preview as Persistent Panel (Item 15)

### Problem

`dbt show` results open as a `WebviewPanel` (editor tab) that gets replaced per model. User wants it in the bottom Panel area like the lineage view.

### Design

Convert to `WebviewViewProvider` pattern:

- New view type: `dbtCoreTools.previewView`
- Register in `package.json` under `views.dbtCoreToolsPanel` alongside lineage
- Panel name: `"dbt Preview"`
- `showModel` command triggers `dbt show` and posts results to the persistent view
- Ready handshake same as lineage (webview posts `{ type: "ready" }`, extension buffers messages until ready)
- Remove `WebviewPanel` creation logic; the view is always available in the Panel area

**Files changed:**

- `src/features/preview/previewPanel.ts` — rewrite from `WebviewPanel` to `WebviewViewProvider`
- `src/extension.ts` — register the new view provider
- `package.json` — add view contribution

The webview HTML/JS/CSS in `src/features/preview/webview/` remain largely the same; only the hosting mechanism changes.
