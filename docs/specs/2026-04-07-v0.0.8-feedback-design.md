# v0.0.8 Feedback Design

**Issue:** TEAMSchools/dbt-core-tools#11
**Date:** 2026-04-07

## Overview

Addresses user feedback from v0.0.8 across two areas: miscellaneous UX improvements and a significant lineage viewer overhaul covering layout, theming, and interaction behavior.

---

## Misc Changes

### 1. Remove Target Settings Persistence

**Problem:** Selecting a target in the status bar writes to `dbtCoreTools.target` in workspace settings, which is unnecessary and noisy.

**Change:**

- `targetSelector.ts`: store the selected target in a module-level `Map<string, string>` instead of calling `config.update()`
- `getCommandOptions()` in `modelCommands.ts`: read from the in-memory map instead of `vscode.workspace.getConfiguration("dbtCoreTools").get("target")`
- Remove `dbtCoreTools.target` from `package.json` contributes configuration
- Selection resets to profile default on window reload (acceptable tradeoff)

### 2. Remove Compile Progress Notification

**Problem:** The `vscode.window.withProgress(ProgressLocation.Notification)` dialog during auto-compile is redundant — the manifest status bar item already shows a spinning animation with the compile label. When minimized, the notification duplicates the status bar.

**Change:**

- `compiledSql.ts`: remove the `withProgress()` wrapper around the compile call
- Keep `manifestStatus.setRunning()` / `clearRunning()` calls — the status bar animation is sufficient feedback

### 3. Faster Compiled SQL Update-After-Save

**Problem:** On save, the cycle is `dbt parse` → wait → `dbt compile -s <model>` — two sequential dbt process startups. The parse-then-compile chain is the main latency source.

**Change:**

- `parseOnSave.ts`: when a compiled SQL document is open for the saved model, skip `dbt parse` and go straight to `dbt compile -s <model>`
- `dbt compile` implicitly parses the project and writes to `manifest.json`, so the manifest watcher picks up the updated `compiled_code`
- When no compiled SQL doc is open, continue running `dbt parse` on save as before (current behavior)
- Cancel any in-flight parse before spawning compile (same as current cancel logic)

### 4. Extension Marketplace Icon

**Problem:** No extension icon configured for the VS Code marketplace/sidebar.

**Change:**

- Convert `media/dbt-icon.svg` to a 128x128 PNG as `media/dbt-icon.png`
- Add `"icon": "media/dbt-icon.png"` to `package.json`

### 5. Shrink File Icon in Explorer

**Problem:** The dbt icon for `.sql` files is too large in the file explorer tree.

**Change:**

- Adjust `media/dbt-icon.svg` internal dimensions — add padding within the viewBox so the visible icon area is smaller relative to the bounding box (e.g., expand viewBox from `0 0 32 32` to `0 0 48 48` while keeping the path centered at 32×32)

## Lineage Changes

### 6. Drop Dagre, Custom Layout Engine

**Problem:** Dagre re-layouts the entire graph on every change, causing large jumps. Users want nodes to stay in place when expanding. ELK.js was considered but its `noLayout` pinning doesn't account for pinned nodes as obstacles — expanding upstream from a previously-expanded child can overlap with existing nodes in the same column.

**Change:**

- Remove `@dagrejs/dagre` dependency from `package.json`
- Replace `layout.ts` with custom positioning logic:
  - **Initial layout:** assign each depth level an x-column offset, stack nodes vertically within each column with spacing
  - **Expand:** position new nodes adjacent to the expanded node (centered on parent's y), then run collision resolution against **all** nodes (not just new ones) to nudge overlaps apart
  - **Collision resolution:** after any position change, detect vertical overlaps at each depth column and shift nodes apart by the minimum distance needed. Adapted from ReactFlow's documented collision resolution pattern. Checks all nodes equally — no pinning distinction.
- Tighter default spacing than current dagre config (`nodesep: 30`, `ranksep: 80`)
- Sibling nodes offset/staggered to conserve vertical space

### 7. Lock Defaults OFF, New Center/Reset Behavior

**Problem:** Lock defaults ON, which means the graph doesn't update on file focus changes. Reset behavior is too aggressive — expanding/collapsing rebuilds the entire graph.

**New behavior:**

| Action                       | Effect                                                                                                                              |
| ---------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| File focus change (unlocked) | Update `currentNodeId` highlight, recenter viewport. **Do not rebuild graph or remove expanded nodes.**                             |
| File focus change (locked)   | No change (current behavior)                                                                                                        |
| Node body click              | Opens file in editor (current behavior). The file focus change handler then recenters viewport. No separate recenter action needed. |
| Expand button (+)            | Add new depth-1 nodes from that node. Existing nodes stay in place.                                                                 |
| Collapse button (−)          | Remove the expanded nodes for that direction only. Do not reset entire graph.                                                       |
| Reset button                 | Rebuild graph to depth-1 from current center node.                                                                                  |
| Right-click → Reset          | Same as reset button.                                                                                                               |

- Default `locked` state: `false` (was `true`)
- `updateCenter` message: updates highlight + viewport only (no graph rebuild)
- `resetCenter` message: full graph rebuild (only from reset button / right-click reset)
- Collapse sends a targeted removal message instead of triggering `resetCenter`

### 8. Fix Expand/Collapse Button State

**Problem:** Expanded nodes are not contracting and the button is not changing from + to −.

**Root cause:** Collapse currently triggers a full `resetCenter`, which clears the expanded state tracking. With the new targeted collapse behavior (item 7), the expand/collapse state will be tracked correctly per node per direction.

**Change:**

- Collapse sends a new `collapseDirection` message type that removes only the nodes added by that expansion
- `expandedRef` map correctly tracks per-node, per-direction state
- Button renders `−` when `expandedUpstream`/`expandedDownstream` is true, `+` otherwise

### 9. Dark Theme + Dot Background

**Problem:** Current theme uses VS Code theme vars with transparent node fills. User wants ReactFlow dark theme defaults with dot background.

**Change:**

- Add `<Background variant="dots" />` ReactFlow component
- Background color: `#1a1a2e`
- Solid dark fills per resource type (see color palette below)
- ReactFlow dark theme defaults for edges, controls, and chrome
- Current node indicator: box-shadow glow (`0 0 8px rgba(border-color, 0.3)`) instead of thick border
- Edge color: use ReactFlow dark theme default

### 10. Color Palette (4 Groups)

| Group  | Resource Types                   | Border    | Fill      |
| ------ | -------------------------------- | --------- | --------- |
| Blue   | model                            | `#64b5f6` | `#1b2a3e` |
| Teal   | source, seed                     | `#4ecdc4` | `#1b3a36` |
| Amber  | snapshot                         | `#ffcc80` | `#2e2518` |
| Purple | exposure, metric, semantic_model | `#ce93d8` | `#2a1b2e` |

Fallback for unknown types: border `#6e6e6e`, fill `#2a2a2a`.

### 11. Fixed Node Size + Badge Redesign

**Problem:** Nodes vary in size. Contract uses emoji shield (🛡). Test badge is a yellow circle positioned outside the node. Badges are not aligned.

**Change:**

- All nodes: fixed **160×64px** regardless of resource type or badge presence
- Badges rendered as a horizontal flex row inside the node, below the materialization label
- Badge row: fixed **16px height**
- **Contract badge:** codicon `$(shield)` in a pill, colored to match the node's resource-type color
- **Test badge:** codicon `$(beaker)` + count in a pill, colored to match the node's resource-type color
- Badge background: 25% opacity of border color
- Badge border: 1px solid border color
- Both badges same vertical size, horizontally aligned

### 12. Tighter Node Spacing

**Problem:** Nodes are too spread out, wasting vertical space.

**Change:**

- Reduce horizontal gap between depth columns (was `ranksep: 80`)
- Reduce vertical gap between nodes in the same column (was `nodesep: 30`)
- Specific values to be tuned during implementation, starting with approximately `rankGap: 60`, `nodeGap: 20`
- Sibling nodes staggered/offset vertically to conserve space
